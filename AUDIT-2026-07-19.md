## 3. Findings

### F1 — Multi-episode ranges rejected (BY DESIGN — no fix requested)

`internal/naming/naming.go:240`

```go
if hasUnsupportedMultiEpisodeRange(baseName) {
    return nil, fmt.Errorf("%w: unsupported multi-episode range in: %s", ErrParseFailed, filename)
}
```

Introduced deliberately in `192af25f` (2026-07-10): *"Parse now fails cleanly on
multi-episode ranges instead of mis-filing"*. Locked in by
`TestParseTVShowFromPath_RejectsUnsupportedMultiEpisodeRange`.

Consequence: 16 *Winnie the Pooh* files (`S02E01E02` … `S04E10E11`, ~28 GB)
failed 8× each, were negative-cached (`Hydrated negative cache from prior
failures | entries=16`), and sat in `complete/` from 07-13 to 07-19.

Two filename forms exist in the wild:
- `S02E01E02` — concatenated (the Pooh releases)
- `S01E061-062-063-064` — hyphenated (already the test fixture)

**Owner's decision: NOT worth fixing.** Judged an aberration in naming
conventions affecting <0.1% of releases. The 16 files were deleted on 07-19.
Recorded here only so the guard's behaviour isn't mistaken for a regression.

*Secondary observation:* AI enhancement is enabled
(`model=gemma4:31b-cloud`) but did not engage on these parse failures —
the activity log shows `parse_method: regex`, `deterministic: true`. Worth
confirming whether the AI fallback is intended to cover `ErrParseFailed`.

### F2 — Jellyfin integration silently disabled for process lifetime (HIGH)

```
Jul 16 12:21:17 [WARN] [daemon] Jellyfin connection failed, disabling integration |
  error=executing request (attempt 3/3): Get "http://localhost:8096/System/Info":
  dial tcp [::1]:8096: connect: connection refused
```

Boot-order race: p2j probed Jellyfin before the container was listening, exhausted
3 retries, and disabled the integration **permanently for the process lifetime**.
Verified zero refresh/notify calls in the 2+ days since; 44 files were organised in
that window with Jellyfin never told to scan.

Jellyfin was healthy at audit time (HTTP 200, v10.11.11) — only p2j's in-memory
flag was stuck off.

**Proposed fix (code):** replace the one-shot startup probe with a periodic
reconnect, or re-probe lazily before each notify rather than latching a boolean
at startup. A startup probe failure should degrade, not permanently disable.

**Proposed fix (ops, no code):** add ordering to
`/etc/systemd/system/plex2jellyfin-daemon.service`:

```ini
After=docker.service
ExecStartPre=/bin/sh -c 'until curl -sf http://localhost:8096/System/Info/Public >/dev/null; do sleep 2; done'
```

Not applied — requires root. See §5.

### F3 — "Recently Added" broken by mtime preservation (HIGH)

**This is the root cause of new content never appearing in Recently Added.**

`internal/transfer/rsync.go:157-161`

```go
func (r *RsyncTransferer) buildArgs(opts TransferOptions, removeSource bool) []string {
    args := []string{
        "--progress",
        "--partial",
        "-a",          // <-- archive implies -t (preserve times)
    }
```

`-a` preserves source mtimes. Release groups stamp mtimes at encode time, so
BluRay rips arrive with years-old mtimes and keep them through the move.
Jellyfin derives `DateCreated` from mtime, so freshly-added content is filed as
years old and never surfaces in Recently Added.

Evidence — The Last Ship S05E08:

| Field | Value |
|-------|-------|
| ctime (actually written to library) | 2026-07-19 00:42 |
| mtime (preserved from rip) | 2021-08-03 |
| Jellyfin `DateCreated` | 2021-08-02 |

All 33 episodes are present and playable — purely a visibility/ordering defect.

**Ruled out — no Jellyfin-side fix exists.** Jellyfin 10.11.11 exposes no
date-added behaviour setting: absent from `/System/Configuration` and from all
42 keys of `LibraryOptions`. (That option is an Emby feature; Jellyfin does not
have it.) An earlier hypothesis that this was a one-setting dashboard fix was
**wrong** — the fix must be file-side.

**Proposed fix (code):** `buildArgs` already has a selective-disable branch at
line 180:

```go
if !opts.PreserveAttrs {
    args = append(args, "--no-owner", "--no-group", "--no-perms")
}
```

Adding `--no-times` here (or introducing a `PreserveTimes` option defaulting to
false) follows the established pattern and is a ~1-line change. Consider whether
this belongs behind a config key, since some users may *want* original mtimes.

**Backfill caveat:** this fixes new content only. Already-ingested items keep
their stored `DateCreated`. A global metadata refresh would restamp *everything*
to now and flood Recently Added for all 5 users — **do not do that**. If the
existing backlog matters, remediate narrowly (touch specific files + targeted
refresh), not library-wide.

### F4 — Episode titles stripped on rename (MEDIUM)

**100% of TV successes affected — 71 of 71.**

p2j writes `Lucky (2026) S01E02.mkv`, discarding the episode title present in the
source filename. Harmless while TMDB has metadata; degrades badly when it doesn't:

| Episode | Source filename | Jellyfin shows |
|---------|-----------------|----------------|
| S01E01 | `Lucky.2026.S01E01.No.Shortcuts.1080p...` | `No Shortcuts` ✓ |
| S01E02 | `Lucky.2026.S01E02.1080p.WEB.h264-ETHEL` | `Lucky` ✗ |

The series matched correctly (TMDB `278624`, IMDb `tt34866681`). E02 fails purely
because TMDB hasn't populated that episode yet (show premiered July 2026) and the
filename retains no title to fall back on.

This is the user-reported *"strange labelling for titles like Lucky"* symptom.

**Proposed fix (code):** emit Jellyfin's recommended form
`Show (Year) S01E02 - Episode Title.mkv` when a title is parsed from the source.
Jellyfin's naming guidance specifies this precisely as the metadata-gap fallback.

### F5 — Season packs containing no episode-numbered files retry forever (MEDIUM)

`internal/organizer/organizer.go:815-852`

```go
tv, err := naming.ParseTVShowFromPath(path)
if err != nil || tv.Season <= 0 || tv.Episode <= 0 {
    result.Unresolved = append(result.Unresolved, path)
    continue
}
...
if len(result.Imported) == 0 {
    result.Skipped = true
    result.SkipReason = "season_pack_unresolved"
    result.Error = fmt.Errorf("season pack has no episode-numbered video files: %s", releaseDir)
```

Trigger: `The.Last.Ship.S02.BONUS.2015.BluRay.1080p.AC3.x264-MTeam-Obfuscated` —
a single 2.18 GB BluRay **extras disc**. Parses fine as a season pack
(`title=The Last Ship, season=2`) but contains no episode number, so
`Episode == 0` → unresolved → `Imported` empty → skipped. Retried on every
periodic scan (07-18 at 07:30, 08:01, 10:06) indefinitely.

The skip logic is **correct** — refusing to guess beats mis-filing. The real gap
is that p2j has no concept of *extras/bonus* content, so such releases can never
resolve and loop forever.

**Proposed fix (code):** recognise extras markers (`BONUS`, `EXTRAS`,
`BEHIND.THE.SCENES`, `FEATURETTE`) and route to Jellyfin's extras convention
(`<Series>/extras/`, or the `-behindthescenes` suffix). Failing that, mark
terminally-unresolvable packs so they stop being retried.

**Already handled manually** — see §5.

### F6 — Duplicate inotify events produce phantom failures (LOW)

Both "transfer failed" errors were the same file processed twice:

```
14:36 S04E10.Endgame ... success  -> /mnt/STORAGE6/.../Season 04/The Last Ship S04E10.mkv
14:36 S04E10.Endgame ... failed   (rsync: source file not found)
14:38 S05E08.Honor   ... success  -> /mnt/STORAGE6/.../Season 05/The Last Ship S05E08.mkv
14:41 S05E08.Honor   ... failed   (rsync: source file not found)
```

The first attempt succeeded and moved the source; the duplicate event then found
it gone. **No data loss** — both files verified present in the library.

**Proposed fix (code):** treat "source missing AND target already exists" as
already-done rather than a failure, so it stops polluting the failure metrics.
